# grill-with-docs — 想法打磨（有代码库）

## Goal

将用户的模糊想法通过多轮追问打磨清晰：
1. 识别模糊术语和隐藏假设
2. 探索代码库验证假设
3. 产出 GRILL-NOTES.md（临时文件）
4. 停下让用户确认，同意才进 `propose`

## Workflow

### 第一步 · 加载上下文

1. 读 `docs/andao_specs/CONTEXT.md`（了解领域语言和既有抽象）
2. 读 `docs/andao_specs/ARCHITECTURE.md`（如存在）
3. 问用户一句话描述想法

### 第二步 · 多轮追问

每轮最多 3 个问题。追问方向：

| 方向 | 问题示例 |
|------|---------|
| 目标 | "这个功能解决了什么问题？" |
| 用户 | "谁会使用这个功能？" |
| 边界 | "这个功能不包括什么？" |
| 依赖 | "依赖现有的哪个模块/抽象？" |
| 假设 | "你认为用户登录已经存在，对吗？" |
| 优先级 | "第一期必须有什么？什么可以后续再做？" |

**规律**：每轮至少一个问题是针对"你刚才说的是什么意思？"（识别模糊术语）

### 第三步 · 代码库验证

当用户提到以下内容时，探索代码库验证：

| 用户说 | 探索操作 |
|--------|---------|
| "类似已有的 X" | grep 查找 X 的实现 |
| "在登录模块上加" | 检查登录模块是否存在 |
| "用户已经能做了" | 验证该能力是否存在 |
| "加一个接口" | g-rep 检查是否有类似接口 |

### 第四步 · 产出 GRILL-NOTES.md

在 `docs/andao_specs/<auto-generated-id>/`（如果已有活跃 change 则使用其 id）下产出 `GRILL-NOTES.md`：

```markdown
# Grill Notes — <id>

## 术语（待验证）
- <术语>：用户说的大概含义
  - 验证问题：<需要确认的问题>

## 冲突假设
- <假设 A> vs <假设 B>
  - <需要解决哪个？>

## 待验证
- [ ] <验证项 1>
- [ ] <验证项 2>
```

**注意**：GRILL-NOTES.md 中的所有术语尚未验证，**不写入 CONTEXT.md**。

### 第五步 · 停下确认

输出：
```
想法已打磨完成（<X 轮追问>）。
核心需求：<一句话总结>
未确认项：<X 项>

要继续进入 propose 产正式提案吗？
1. 继续（推荐）
2. 继续打磨
3. 不做了
```

用户选 1 → 路由到 `propose`。
用户选 2 → 回到第二步继续追问。
用户选 3 → 清空临时 GRILL-NOTES.md，回到 `ask-andao`。

## 约束

- 禁止直接将术语写入 CONTEXT.md（由 `model` 验证后写入）
- 每轮最多 3 个问题
- 至少追问 2 轮后才可以"停下确认"
